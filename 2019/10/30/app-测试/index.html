一、环境配置

1.运行环境：

  • JDK ：java环境就不用说了，android都包含java外壳。
  • ADT ：ADT 中除了 adb（Android Debug Bridge）和 Monitor，还包含了 Emulator 虚拟机和Eclipse 开发环境；
  • NDK ：NDK 可以在 Windows 上编译 Android(arm)使用的 Native 层可执行文件（C/C++），各项 Native 层测试工具都可用 NDK 制作；
  • GDB ：调试必备。

2.功能工具：
web类：

  • BurpSuite/Fidder：主要用于HTTP的测试操作
  • wireshark：遇到不走HTTP的app，可以用它来定位代码。
    逆向类：
  • IDA ：一般用于处理存在Native层的app。
简单的工具介绍：

1.创建和使用Emulator虚拟机：
Tools->Manage AVDs->Create：

上图红框中部分为建议参数，其余部分可以根据需要进行修改。
之后选中刚刚选中的设备，点击Start->Launch，即可启动虚拟机。 

Emulator虚拟机启动后会自动连接ADB，无需开启USB调试。   
Emulator虚拟机的ADB默认具有root权限，这样可以轻易的完成root。   

替换su：  
adb shell mount -o remount,rw -t yaffs2 /dev/block/mtdblock3 /system    
adb shell cat /system/bin/sh > /system/bin/su    
adb shell chmod 4755 /system/bin/su

2.ADB常用指令：
设备操作类：

  • 查看设备列表：
    adb devices
  • 执行其他命令时选择目标设备：
    adb -s 设备名称 其他命令
    如果你同时连接了多个设备，要选择操作的目标设备，否则会出错。

文件/app操作类：

  • 从安卓设中读出文件：
    adb pull 安卓设备路径 主机路径
  • 向安卓设备中写入文件：
    adb push 主机路径 安卓设备路径
  • 直接安装apk到安卓设备中:
    adb install APK文件在主机中的路径
  • 卸载安卓设备中的app
    adb uninstall APP包名

系统命令类：

  • 打开安卓设备的交互式shell
    adb shell
  • 在安卓设备中执行命令：
    adb shell 要执行的命令。

二、客户端静态安全

1.安装包签名：

风险描述
在 Android 中，包名相同的两个 APK 会被认为是同一个应用。当新版本覆盖旧版本时，签名证书必须一致，否则会被拒绝。（即使开启了“允许未知来源的应用”）。如果 APK没有使用自己的证书进行签名，将会失去对版本管理的主动权。

风险等级 ：
中风险

测试方法：
➢ 方法 1：使用 Android SDK 中的 apksigner.jar 检查安装包的签名：
java -jar .\apksigner.jar verify -v –print-certs APK 文件路径
➢ 方法 2：ApkBurster：
加载 APK 后，会自动输出安装包中 classes.dex 的签名。
风险分析：
只有在使用直接客户的证书签名时，才认为安全。Debug 证书、第三方（如开发方）证书等等均认为风险。缺少 V2 签名，或存在未受签名保护的文件时，均认为风险。常见的不合格情况包括：

  • 未签名或签名无效：
  • 缺少V2签名：
  • 使用Debug证书签名：
  • 使用个人证书/开发方证书签名：
  • 存在未受签名保护的文件：

2.修复方案
建议客户端使用 APP 从属方证书进行签名后进行发布而不是使用第三方开发商的证书进行签名，以防开发商内部监管异常，证书滥用的情况出现。
此外，如果 APK 支持 Android7.0 及以上的系统，建议包含 V2 签名；如果 APK 不支持 Android6.0 及以下的系统，可以考虑不进行 V1 签名。

注意事项：

  • 1.大部分情况下，只要与用户说明相关危害即可，如果客户认为当前证书没有问题的话，可以接受风险。
  • 注意：旧式的验证方法是用 JDK 中的“jarsigner.exe -verify APK 文件路径 -verbose -certs”命令，但此种方式只能验证 V1 签名（而忽略 V2 签名状态）。V1签名方式已被认为不安全，详情请搜索“Janus”漏洞。
    • a) 但 Android 7.0 之前（不含 7.0，即 API Level<24）的系统不支持 V2 签名方式，因此大多 APK 建议选择 V1 和 V2 方式共存的签名方法。
2.客户端数据安全：

风险描述：
视具体的泄露信息而定，敏感信息泄露可能会方便恶意程序窃取凭据，或者泄露一些原本不希望被用户看到的内容。

风险等级：
中风险

测试方法：
Monitor.bat工具（ADT目录下\sdk\tools\下）

修复方案：
正常的文件权限最后三位应为空（rw-rw—-），目录则允许增加一个执行位（rwxrw-x）。

注意事项
1.app的数据文件目录一般为：/data/data/{APP包(package)名}/。
2.app通常需要登陆并使用一段时间后，才会产生足够多的数据文件。

3.本地文件内容安全

风险描述：
检查app数据目录中的文件是否包含敏感内容。包括文本文件、二进制文件、sharedPreference等XML文件、WebView等数据库文件等等。
也需要视具体情况而定，敏感信息泄露可能会方便恶意软件程序窃取凭证，或者泄露一些原本不希望被用户看到的内容。

风险等级：
视具体的泄露信息而定。
高风险：有认证凭据、交易信息、SessionID、通信密钥等高敏感的信息。
中风险：其他非高风险的敏感信息泄露。

测试方法：
依然是使用上面的Monitor.bat工具（ADT目录下\sdk\tools\下）
将安卓设备中的文件复制到有鼠标的主机上，方便查看。

修复方案：
尽量避免在文件、数据库等位置写入敏感信息，如果需要存储，应当进行加密。
注意：
遇到.bd文件大多数为SQLite数据库，使用Navicat等工具方可打开。
